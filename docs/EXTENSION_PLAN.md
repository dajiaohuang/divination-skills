# divination-skills 扩展功能计划

状态：Implemented（technical scope；production release gated）

基线日期：2026-07-23

起始基线：10 个体系、14 个 Skill、54 条结构化规则、技术完整度 10/10

完成快照：10 个体系、30 个 Skill、101 条结构化规则、850 个扩展回放案例、1497 项测试

## 1. 目标

下一阶段不再以“增加更多体系”为主要目标，而是把现有体系从基础垂直切片扩展为可组合的完整产品能力。

本计划首先对标两个被忽略的本地参考：

- `vedic-astro-skills`：参考其计算、读盘、核心分析、专项报告、合盘、校时和即时占问的职责拆分。
- `iztro`：参考其紫微排盘深度、运限层级、宫星查询、四化分析、配置和国际化覆盖。

“对标”只表示建立能力矩阵。项目不会复制参考仓库的代码、提示词、规则资料、翻译或数据表，也不会把它们变成运行依赖。

## 2. 不可破坏的约束

1. 不添加 Git submodule。
2. 不把其他 Git 仓库作为运行、构建或测试依赖。
3. `references/upstream/` 继续由 Git 忽略，参考内容不得进入 Skill 包。
4. `iztro` 只用于人工差异比较，不在生产代码中导入、调用、封装或转译。
5. `vedic-astro-skills` 的 Skill、Prompt 和资源具有附加非商业限制，只能研究架构，禁止复制到 Apache-2.0 项目。
6. 所有新增实现必须有独立来源台账、流派声明、规则 ID、Golden Cases 和发布审核。
7. 计算事实、派生规则、语言解释和用户背景必须继续分层。
8. 出生信息、关系资料和人生事件必须遵守敏感数据与隐私政策。
9. 没有经过专家审核的解释能力只能标记为实验性，不能通过自动生成签署绕过发布门禁。

## 3. 对标结论

### 3.1 `vedic-astro-skills` 的能力结构

参考版本：`e6d3d39073baca87f8e540b9b92fa758a1f5085a`

| 参考 Skill | 主要职责 | 本项目可吸收的架构思想 |
|---|---|---|
| `vedic-calculator` | 原生计算并生成统一结构数据 | 计算器作为主数据源；输出稳定数据契约 |
| `vedic-reader` | 读取 PDF、截图和文本并交叉验证 | 外部盘导入与计算结果分离；差异显式记录 |
| `vedic-core` | 核心盘面审计、分盘交叉和完整报告 | 基础解释与专项产品共享同一事实、规则层 |
| `vedic-career` | 职业专项报告 | 专项报告只选择相关事实，不另造计算口径 |
| `vedic-love` | 关系模式和时机专项报告 | 静态模式与动态时机分开 |
| `vedic-synastry` | 双盘关系分析 | 双方盘独立验证；关系事实具有方向性 |
| `vedic-rectifier` | 根据事件反推出生时间 | 候选时间扫描、反证和样本外验证 |
| `vedic-prashna` | 即时占问 | 即时盘与本命盘隔离，可选交叉只读 |

值得采用的原则：

- `calculator → validator/importer → core → specialist` 的分层。
- 统一结构数据是各 Skill 之间唯一可靠接口。
- 时间精度决定允许使用的计算层和解释层。
- 专项报告共享事实和规则，不复制一套独立知识库。
- 合盘、校时、即时占问使用独立数据契约，避免污染本命盘。
- 长报告按阶段生成，并在每一阶段保留证据与限制。

不直接采用的部分：

- 不复制其八个 Skill 的提示词、资源文件或报告文案。
- 不采用其依赖组合或 Swiss Ephemeris、PyJHora、dashaflow 等实现作为运行前提。
- 不采用超过一千行的单体 Skill 指令；本项目继续使用短 `SKILL.md`、确定性脚本和按需参考文件。
- 不把“验前事命中”作为精度证明；校时必须有保留集、反证和不确定性输出。
- 不承诺固定事件、精确恋爱日期或出生时间可被稳定校准到某一分钟。

### 3.2 `iztro` 的能力结构

参考版本：`2.5.8`，提交 `9d39f1743bf31c2b3c635c9b9556215d9c90ee2c`

| 能力域 | `iztro` 参考能力 | 当前项目状态 | 差距 |
|---|---|---|---|
| 输入 | 公历/农历、闰月、13 时辰索引 | 公历当地时间、13 时辰索引 | 缺少原生农历输入和显式闰月策略 |
| 边界配置 | 年界、运限年界、生日/自然年、晚子时 | 固定项目策略 | 缺少版本化可选政策 |
| 流派 | 通行排法、中州派、天/地/人盘 | 单一项目基础线 | 缺少独立流派模块和差异矩阵 |
| 本命星曜 | 主星、辅星、煞星、杂曜及多组十二神 | 14 主星、4 辅星 | 星曜目录和定位规则差距最大 |
| 星曜属性 | 亮度、五行、阴阳、四化、作用层级 | 生年四化基础标签 | 缺少亮度、类别和动态层级 |
| 宫位 | 命身宫、来因宫、宫干支、五行局 | 命身宫、宫干支、五行局 | 缺少来因宫等扩展字段 |
| 运限 | 大限、小限、流年、流月、流日、流时 | 大限范围 | 缺少完整动态盘 |
| 分析器 | 宫位/星曜查询、三方四正、空宫、四化飞宫和自化 | 结构报告 | 缺少可复用查询层 |
| 扩展性 | 配置四化/亮度、插件 | 固定实现 | 需要版本化配置，不照搬运行时插件 |
| 国际化 | 简中、繁中、英文、日文、韩文、越南文 | 以中文结构数据为主 | 缺少术语 ID 与翻译层 |
| 测试 | 多模块单元测试和公开 API 测试 | Golden、边界、争议和打包测试 | 本项目治理更强，但功能覆盖更浅 |

应采用的方向：

- 把“排盘”和“查询分析器”拆成两个项目自有模块。
- 为本命、大限、小限、流年、流月、流日、流时统一定义 `scope`。
- 所有宫位、星曜、四化和运限对象使用稳定、不依赖语言的内部 ID。
- 流派差异以配置和独立规则包实现，不通过全局可变状态静默切换。
- 多语言只翻译显示名和解释模板，不翻译内部 ID。

不直接采用的部分：

- 不依赖 npm 包 `iztro`。
- 不复制其 TypeScript、星曜常量表、测试预期或翻译。
- 不把插件注入作为第一阶段目标；先使用可验证的静态注册表。
- 不以和 `iztro` 完全一致作为正确性的唯一标准。差异必须按流派、历法、闰月和边界政策分类。

## 4. 目标产品架构

```text
raw input / imported chart
        |
        v
calculator -----> validator -----> canonical chart
                                       |
                       +---------------+---------------+
                       |               |               |
                       v               v               v
                     core          timing engine    comparison engine
                       |               |               |
                       +-------+-------+-------+-------+
                               |
                               v
                career / relationship / synastry / Q&A
```

每个确定性体系逐步具备以下模块，但不要求每个模块都是独立 Skill：

1. `calculator`：只产生计算事实。
2. `validator`：校验输入、Schema、版本、来源和政策。
3. `importer/reader`：将外部结构化盘映射为项目 Schema，并保存差异；第一阶段不做图片 OCR。
4. `core`：只解释已经验证的结构事实。
5. `timing`：计算运限、行运或时间窗口，不直接生成结论。
6. `comparison`：合盘、双盘或多盘事实层。
7. `specialist report`：职业、关系等报告视图；共享核心规则。
8. `rectifier`：仅适用于时间敏感的确定性体系，并必须输出候选区间和不确定性。

### 4.1 新增公共契约

| 契约 | 用途 |
|---|---|
| `reading-session.schema.json` | 问题、盘引用、用户上下文、同意状态和数据分类 |
| `chart-import.schema.json` | 外部盘来源、字段映射、缺失项和冲突项 |
| `confidence-profile.schema.json` | 输入精度、计算覆盖、解释允许层级和降级原因 |
| `timeline.schema.json` | 大运、流年、行运等统一时间段 |
| `comparison.schema.json` | A/B 双盘引用、方向性事实和共同时间窗口 |
| `report-profile.schema.json` | core、career、relationship 等报告章节选择，不承载规则本身 |

这些契约必须保持体系中立；具体计算字段仍由各体系 Schema 定义。

## 5. 扩展路线图

工期是单人全职开发的建议量级，不是发布日期承诺。

### M6：跨体系会话、置信度和时间线契约

建议工期：2 周

交付：

- 六个公共契约及示例。
- 版本迁移策略和兼容性测试。
- 输入时间精度等级：精确、分钟区间、小时区间、日期级、不确定。
- 根据精度自动关闭高敏感计算层。
- 用户问题分类：本命、时间、关系、职业、即时占问。
- 报告章节选择器，不新增领域结论。

验收：

- Schema 正反例全部通过。
- 同一输入得到稳定会话 ID 和事实引用。
- 敏感出生数据默认不进入日志、构建产物或测试快照。
- 所有现有 14 个 Skill 可继续读取旧输入，或通过显式迁移读取。

### M7：原生紫微本命盘 v0.2

建议工期：4 周

目标是缩小与 `iztro` 的基础排盘差距，但采用项目独立实现。

交付：

- 公历和农历输入、闰月标识、闰月处理策略。
- 年界、晚子时和大限边界的版本化政策。
- 扩充辅星、煞星、桃花星、解神、禄存、天马和杂曜目录。
- 长生十二神、博士十二神、岁前十二神、将前十二神。
- 星曜类别、五行、阴阳和亮度数据契约。
- 小限年龄表、来因宫等本命结构字段。
- 默认项目流派与中州候选流派完全隔离；未完成时只交付默认线。
- 新增 `ziwei-validator` Skill。

规则族建议：

- `ZIWEI-INPUT-*`
- `ZIWEI-PALACE-*`
- `ZIWEI-STAR-MAJOR-*`
- `ZIWEI-STAR-MINOR-*`
- `ZIWEI-STAR-AUX-*`
- `ZIWEI-BRIGHTNESS-*`
- `ZIWEI-CYCLE-*`
- `ZIWEI-BOUNDARY-*`

验收：

- 至少 100 个本命标准盘。
- 至少 30 个闰月、立春、晚子时、日期上下界案例。
- 至少 20 个流派或参考实现差异案例。
- 每颗星的定位规则至少有直接单元测试和一个 Golden Case。
- 可生成与 `iztro` 的本地差异报告，但测试和发布包在没有参考仓库时仍完全通过。
- 差异报告将每项差异分类为项目错误、参考错误、流派差异、边界政策差异或待研究。

### M8：紫微运限和查询分析器 v0.3

建议工期：4 周

交付：

- 大限、小限、流年、流月、流日、流时统一动态层。
- 各动态层的宫位名称、天干地支、四化和流曜。
- 宫位查询、星曜查询、三方四正和空宫判定。
- 生年四化、宫干飞化、自化和四化落宫事实。
- 静态注册式模式检测器，不开放任意代码插件。
- 新增 `ziwei-timing` Skill；只返回时间结构与证据。

验收：

- 十二宫旋转和三方四正进行穷举属性测试。
- 任意星曜只能存在于允许的作用层级。
- 四化查询必须能回溯到发出宫、目标星和目标宫。
- 运限边界前后各一秒、一天和一个月都有明确测试。
- 对同一目标日期重复计算必须字节级一致。

### M9：紫微解释层 v0.4

建议工期：3 周开发，加独立专家审核时间

交付：

- `ziwei-core`：本命结构解释。
- 把宫位、星曜、三方四正和四化规则写成独立规则文件。
- 解释结论必须引用事实 ID、规则 ID 和来源 ID。
- `ziwei-reader` 只导入结构化 JSON；PDF、截图 OCR 延后。
- 简体中文术语 ID 与显示层分离，为繁中和英文预留。

验收：

- 至少 50 个专家候选分析案例。
- 未完成专家接受和领域签署前，`ziwei-core` 只能以 experimental 状态构建。
- 引用完整率和事实引用有效率均为 100%。
- 未知规则误用率和跨体系规则污染率均为 0。
- 不输出固定人生事件、身份诊断或确定性吉凶。

### M10：把 Vedic 八角色架构应用到八字和西方占星

建议工期：6 周

#### 八字

新增候选能力：

- `bazi-reader`：导入外部四柱 JSON/文本，逐字段对照原生计算。
- `bazi-timing`：大运、流年、流月和规则激活事实。
- `bazi-synastry`：双方四柱独立验证后生成方向性关系事实。
- `bazi-rectifier`：候选时辰/时间区间扫描，默认只到时辰级，不承诺分钟精度。

#### 西方占星

新增候选能力：

- `western-validator`：独立校验外部本命盘。
- `western-timing`：行运、太阳返照；推运必须作为后续独立流派。
- `western-synastry`：双盘相位和宫位触发。
- `western-reader`：第一阶段只接受结构化 JSON/CSV。
- `western-rectifier`：最后实施，必须有事件保留集。

验收：

- 导入器永远不能覆盖原生计算，只能生成差异记录。
- 合盘输出明确 `A → B`、`B → A` 和对称事实。
- 时间模块与本命解释规则分离。
- 每个新的解释型 Skill 都需要独立的 50 份专家审核队列。

### M11：职业、关系和问答产品层

建议工期：4 周，加领域审核时间

Vedic 参考仓库把职业和恋爱做成独立 Skill。本项目先把它们实现为共享报告配置，只有在触发测试证明需要独立 Skill 时才拆分。

候选产品视图：

- `career`：职业环境、工作方式、资源与风险，不推荐具体高影响职业决策。
- `relationship`：关系模式、沟通和边界，不推断第三方隐私。
- `timing`：显示结构性时间窗口、证据和反证，不承诺事件一定发生。
- `qa`：只回答当前盘和已审核规则能够支持的问题。

首批适用体系：

1. 八字
2. 西方占星
3. 紫微斗数

验收：

- 同一事实在 core、career、relationship 中使用相同事实 ID 和规则 ID。
- 专项报告不得另设未经登记的计算口径。
- 每个结论同时展示支持证据、反证或限制。
- 医疗、法律、财务、就业录用等高影响问题必须降级为反思性信息。

### M12：关系、校时和即时占问高级层

建议工期：按体系分别评估

交付顺序：

1. 八字、西方占星、紫微的双盘比较。
2. 八字和西方占星的候选时间扫描。
3. 西方卜卦占星的独立体系评估。
4. 周易、六爻、奇门继续作为各自的即时占问体系，不套用 Vedic Prashna 规则。

校时硬门槛：

- 至少五个带日期区间的历史事件。
- 训练事件和保留验证事件分离。
- 输出候选时间区间和排名，不伪造唯一答案。
- 软性性格描述不能打破硬事件平局。
- 如果证据不足，必须返回 `underdetermined`。

### M13：其他体系深度扩展

在前三个计算型体系稳定后再执行：

- 六爻：用神选择流派、旺衰、动变关系和应期作为不同规则包。
- 奇门：天盘、九星、八门、八神、空亡、入墓、击刑和门迫。
- 周易：经典文本来源层、动爻取用流派和文本版本差异。
- Lenormand：牌对、九宫和 Grand Tableau。
- Tarot：更多牌阵、组合关系、日记和统计，但不引入商业牌组素材。
- 卢恩：历史资料层与现代占卜层严格分离。
- 数字命理：Chaldean 作为独立流派；非拉丁姓名必须有明确转写政策。

## 6. 计划中的 Skill 变化

### 第一优先级：确定新增

| Skill | 原因 |
|---|---|
| `ziwei-validator` | 当前计算与校验职责混在一个 Skill |
| `ziwei-timing` | 动态盘需要独立输入、事实和缓存边界 |
| `ziwei-core` | 解释层必须与计算层物理分开 |
| `bazi-timing` | 现有大运只有结构数据，缺少时间规则层 |
| `bazi-synastry` | 双盘关系需要独立数据契约 |
| `western-validator` | 当前缺少独立外部盘验证入口 |
| `western-timing` | 行运与本命计算生命周期不同 |
| `western-synastry` | 双盘相位需要方向性事实模型 |

### 第二优先级：满足触发评估后新增

| 候选 Skill | 拆分条件 |
|---|---|
| `ziwei-reader` | 结构化导入工作流明显不同于 validator |
| `bazi-reader` | 外部盘导入需求足够稳定 |
| `western-reader` | 支持至少两个明确外部数据格式 |
| `bazi-rectifier` | 时辰级扫描通过保留集评估 |
| `western-rectifier` | 事件扫描有独立算法和足够案例 |
| `ziwei-synastry` | 明确主流派和双盘规则来源 |
| `*-career` | 报告配置无法可靠覆盖职业触发和流程 |
| `*-relationship` | 报告配置无法可靠覆盖关系触发和流程 |

不为只有一段提示词差异的功能创建新 Skill。

## 7. 测试与发布门禁

### 7.1 计算扩展

- 每个新增计算模块至少 50 个标准案例。
- 时间或历法模块至少 30 个边界案例。
- 流派配置至少 20 个明确分歧案例。
- 所有随机能力必须支持种子重放。
- Windows 和 Linux 输出必须规范化一致。

### 7.2 解释扩展

- 每条语言结论必须引用事实和规则。
- 每个新解释型 Skill 至少准备 50 个专家候选案例。
- 必须包含反例、输入不足和禁止结论案例。
- 专家接受和领域签署未完成时，发布状态保持关闭。

### 7.3 参考隔离

CI 必须持续断言：

- 没有 `.gitmodules`。
- 没有被跟踪的 `references/upstream/` 文件。
- 生产代码和打包产物没有 `iztro` import、Node 运行时或上游源码。
- Skill ZIP 不包含参考仓库文件名、Git 元数据或来源为 `reference_only` 的内容。
- 没有网络和参考目录时，全部生产测试仍可通过。

### 7.4 对标报告

差异比较工具只在开发者本地运行，输出以下统计：

- 完全一致字段。
- 仅显示名不同字段。
- 流派差异字段。
- 边界政策差异字段。
- 当前项目缺失字段。
- 疑似实现错误字段。

对标报告不是 Golden Set，也不能自动把参考输出写入生产案例。
