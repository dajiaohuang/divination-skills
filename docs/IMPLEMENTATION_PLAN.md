# divination-skills 实施计划

版本：0.1
日期：2026-07-23
状态：既定技术范围已实施；外部人工签署待完成

## 1. 目标与非目标

### 目标

建立一个能够持续扩展的多体系占卜工程，使每个体系都具备：

- 明确的流派、年代、术语和产品边界；
- 可复现的输入规范与确定性计算（适用时）；
- 带来源和置信度的结构化规则；
- Golden Set、边界案例和流派分歧案例；
- 从事实到规则再到解释的可审计输出；
- 可生成 Codex 等不同运行环境产物的 Skill 源文件；
- 版权、隐私和高风险场景的明确约束。

### 当前非目标

- 不在第一阶段同时实现全部占卜体系；
- 不把未经整理的电子书直接作为生产 RAG 语料；
- 不复制 `vedic-astro-skills` 的提示词、资源文件或受限表达；
- 不承诺或宣传未经设计、盲测和复核支持的“准确率”；
- 不用占卜结论替代医疗、法律、财务等专业判断。

## 2. 目标架构

```text
用户输入
  -> 输入规范化
  -> 确定性计算 / 抽取
  -> 事实校验
  -> 规则匹配与冲突裁决
  -> 带证据的派生结论
  -> 用户语言解释与报告
  -> 反馈和评测（不回写原始事实）
```

四类数据必须物理或逻辑分离：

1. `raw_input`：用户原始输入；
2. `normalized_input` 与 `computed_facts`：可重算、可校验的数据；
3. `derived_findings`：规则 ID、输入事实和裁决过程；
4. `narrative`：面向用户的解释、建议和不确定性说明。

建议的正式目录结构：

```text
divination-skills/
├── common/
│   ├── schemas/                 # 跨体系 JSON Schema
│   ├── safety/                  # 高风险、隐私、表达政策
│   ├── report-spec/             # 报告和证据展示规范
│   └── evaluation/              # 通用评测器与量表
├── systems/
│   └── <system>/
│       ├── SCOPE.md
│       ├── LINEAGE.md
│       ├── DATA_CONTRACT.md
│       ├── KNOWN_DISPUTES.md
│       ├── KNOWN_LIMITATIONS.md
│       ├── sources/             # 只存元数据或获授权资料
│       ├── rules/               # 结构化规则与本体
│       ├── calculator/          # 确定性计算代码（适用时）
│       ├── skills/              # 单一源 Skill
│       └── tests/
│           ├── golden/
│           ├── edge_cases/
│           └── disputes/
├── tooling/                     # schema 校验、构建、评测
├── dist/                        # 生成的各平台版本，不手工修改
└── references/upstream/         # 隔离的外部参考仓库
```

## 3. 路线图

时间是单人全职工作的粗略估算；资料授权和专家复核不包含在开发工期内。

### M0：参考基线与工程治理（已完成）

交付物：

- 初始化主仓库；
- 将 `vedic-astro-skills` 作为被主仓库忽略的独立本地克隆，并登记具体提交；
- 登记来源、许可证和可使用边界；
- 写明 clean-room 实现原则；
- 确立 issue、变更记录和决策记录格式。

退出条件：

- 新成员能用一条命令恢复相同参考版本；
- 上游内容不会被误当作本项目可商用源文件；
- 每项外部资料在进入生产目录前都必须有来源台账。

### M1：四个跨体系契约（已完成）

优先创建并冻结 v0.1：

1. `SOURCE_MANIFEST.schema.json`：作者、版本、URL、许可证、商用/改编/翻译/图像权、获取时间、文件哈希；
2. `RULE.schema.json`：规则 ID、体系、流派、条件、结论、例外、优先级、置信度、来源定位；
3. `GOLDEN_CASE.schema.json`：原始/规范输入、中间值、最终盘、必须命中、允许分歧、禁止结论、审核信息；
4. `DISPUTE.schema.json`：冲突主题、各派规则、适用范围、默认裁决和展示方式。

同时实现：

- Schema 示例与正反测试；
- 本地校验命令；
- CI 中的格式、链接和 ID 唯一性检查；
- 最小来源许可检查：生产规则没有合法来源记录时构建失败。

退出条件：

- 每个 Schema 至少有 3 个有效示例和 3 个无效测试；
- 规则可以从来源定位到 Golden Case，再定位到派生结论；
- `main` 分支上的数据契约检查全部通过。

### M2：首个端到端垂直切片（技术实现已完成）

默认选择：**八字，子平法基线**。原因是中文用户价值较直接，同时足以验证历法、流派冲突、规则推导和解释链。若第一目标是最大化现有天文计算复用，则在 M1 结束时可改选西方占星；不要同时启动两套计算体系。

范围先冻结为：

- 本命排盘与基础解读；
- 公历输入、IANA 时区、经纬度和历史时区；
- 精确节气；
- 年/月/日/时柱和藏干；
- 十神、五行、刑冲合害等确定性关系；
- 一套明确的旺衰/格局判读路径；
- 大运排布；流年只做到规则支持的最小范围；
- 暂不做盲派混合、合婚、择日和断具体事件。

先决决策必须写进 `LINEAGE.md`：

- 立春换年；
- 节气换月；
- 子初或子正换日；
- 是否以及何时使用真太阳时；
- 顺逆排运与起运岁数算法；
- 对跨日、夏令时和历史时区的处理。

交付物：

- `bazi-calculator`、`bazi-reader/validator`、`bazi-core` 三层最小 Skill；
- 计算输出 JSON，而非以 Markdown 作为唯一主数据；
- 20 个开发 Golden Cases、10 个边界案例、5 个流派分歧案例；
- 两个独立排盘来源的交叉对照；
- 50 条以内高质量规则，避免先追求数量；
- 一份能展示“事实 → 规则 → 解释”的样例报告。

退出条件：

- 所有确定性字段在 Golden Set 上一致，允许差异必须落入分歧记录；
- 相同输入可重现相同计算事实；
- 每条解释至少引用一个事实 ID 和一个规则 ID；
- 模型不能覆盖或改写计算事实；
- 盲测中不出现跨流派无标注混用。

### M3：首个体系生产化（技术门禁已完成，真实签署待完成）

扩展目标：

- 100 份标准排盘；
- 30 份历法、日期交界、夏令时和历史时区边界案例；
- 20 份流派分歧案例；
- 50 份带完整推理链的专家分析；
- 20 份错误输入与纠错案例；
- 结构化匿名案例反馈，明确区分事前判断与事后复盘。

增加：

- 属性测试和回归测试；
- 报告级证据展示；
- 敏感出生数据的保留和删除策略；
- 质量 Model Card：支持范围、来源、已知限制、评测集和人工复核方法；
- 可安装的 v0.1 Skill 包和版本化迁移策略。

退出条件：

- 计算、规则、叙述三个层级的评测分开通过；
- Golden Set 中不存在未解释差异；
- 专家复核、许可审计和隐私检查有签字记录；
- 失败时系统会降级或拒答，不凭模型记忆补算关键字段。

### M4：第二与第三体系（已完成）

建议顺序：

1. **Tarot（RWS 基线）**：快速验证非确定性抽取、牌阵位置、组合规则、随机性审计和日记体验；
2. **西方占星**：验证星历、宫制、相位容许度、行运、合盘等可计算模块复用。

Tarot 必须区分牌面图像版权、牌义来源和产品解释；商业版本优先使用自绘或明确授权的牌面。西方占星的星历引擎在选型时单独审查 AGPL 与商业许可证。

退出条件：

- 第二个体系证明公共 Schema 无需破坏性修改；
- 体系之间的规则 ID、本体和用户上下文完全隔离；
- 平台构建可以从同一 Skill 源生成至少一种可安装产物。

### M5：扩展波次（波次 A/B 已完成，波次 C 未获授权且不在本轮范围）

- 波次 A：紫微斗数、周易文本占问、六爻；
- 波次 B：奇门遁甲、Lenormand、卢恩符文、数字命理；
- 波次 C：风水、手相、面相；
- Human Design 仅在完成独立知识产权评估或获得授权后立项。

每个新体系必须重新经过：范围冻结 → 许可证审计 → 计算规范 → 小型 Golden Set → 垂直切片 → 专家复核。不得因为已有公共框架而跳过领域验收。

## 4. 工作流与状态门

每条来源和规则采用以下状态：

```text
proposed -> rights_checked -> extracted -> reviewed -> tested -> production
                    \-> rejected
```

进入 `production` 的最低要求：

- 来源许可字段齐全；
- 有精确页码、章节、稳定 URL 或代码版本；
- 至少一名领域审核者确认规则表达；
- 至少一个正例和一个反例；
- 与现有规则的冲突已显式记录；
- 能生成机器可验证的派生记录。

## 5. 测试策略

测试分五层，不能只测最终文案：

| 层级 | 主要问题 | 示例 |
|---|---|---|
| 输入 | 时间、地点和问题是否规范化正确 | DST 重叠时刻必须要求明确 offset |
| 计算 | 确定性结果是否正确且可复现 | 节气交界前后月柱变化 |
| 规则 | 条件、例外和优先级是否正确 | 从格条件未满足时不得套用 |
| 推理 | 是否只用允许的事实和规则 | 不从用户背景反推盘面 |
| 表达 | 是否校准不确定性并遵守安全政策 | 不把建议写成不可避免的事件 |

发布门槛不使用单一“总准确率”。至少分别报告计算一致率、规则命中/误用率、引用完整率、流派污染率、事实幻觉率和人工复核结果。

## 6. 资料收集优先级

每个体系按以下顺序收集，而不是先堆书：

1. 流派范围和术语表；
2. 计算规范与两个独立实现/排盘器；
3. 原典与现代规则来源的许可元数据；
4. 结构化规则和争议矩阵；
5. Golden Cases 与边界案例；
6. 经过匿名化、区分事前/事后的专业案例；
7. 竞品流程与产品交互记录；
8. 面向用户的解释模板。

## 8. 需要在 M1 结束前确认的产品决策

这些决策不会阻塞 M0/M1，但会改变 M2：

1. 首发市场：中文用户优先，还是国际用户优先；
2. 首个体系：默认八字，或改为复用更高的西方占星；
3. 商业模式：闭源服务、开源核心，还是双许可证；
4. 专家审核资源：谁负责流派决策和 Golden Set 签字；
5. 用户数据策略：是否保存出生资料、保存多久、如何导出和删除。

## 9. 成功定义

首个可发布版本不是“已经收集很多资料”，而是：给定一个受支持输入，系统能稳定地产生正确计算事实，明确展示采用的流派与规则，为每个重要结论给出来源链，并在资料不足、规则冲突或高风险问题上正确降级。
